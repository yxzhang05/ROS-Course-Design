## 1. 整体架构

本小车的自主导航系统由以下几个主要模块组成，它们通过 ROS 话题和动作服务协同工作：

```
┌─────────────────────────────────────────────────────────────────┐
│                        导航系统总览                              │
│                                                                 │
│  ┌──────────┐    激光扫描     ┌──────────┐   /amcl_pose        │
│  │  激光雷达 │ ─────────────> │   AMCL   │ ─────────────────>  │
│  └──────────┘  /scan         │  (定位)  │   map→odom TF       │
│                               └──────────┘                     │
│                                    ↑                            │
│  ┌──────────┐   /odom         /initialpose（初始位姿输入）        │
│  │  底盘驱动 │ ─────────────> ─────────────────────────>        │
│  │（里程计） │                                                   │
│  └──────────┘                                                   │
│       ↑ /cmd_vel                                                │
│       │                                                         │
│  ┌──────────────────────────────────┐                           │
│  │            move_base             │                           │
│  │  ┌────────────┐  ┌────────────┐  │                           │
│  │  │  全局规划器 │  │  局部规划器 │  │ <── /move_base/goal      │
│  │  │(navfn等)   │  │  (TEB等)  │  │                           │
│  │  └────────────┘  └────────────┘  │                           │
│  │  ┌────────────┐  ┌────────────┐  │                           │
│  │  │ 全局代价图 │  │ 局部代价图 │  │                           │
│  │  └────────────┘  └────────────┘  │                           │
│  └──────────────────────────────────┘                           │
│                                                                 │
│  ┌──────────────────────────┐                                   │
│  │   多点导航节点            │ ──send_goal──> move_base          │
│  │   (multi_point_navigation│ ──/cmd_vel──>  底盘（原地旋转时）  │
│  │    .py)                  │                                   │
│  └──────────────────────────┘                                   │
└─────────────────────────────────────────────────────────────────┘
```

**数据流顺序（正常导航一次的完整过程）：**

1. 激光雷达持续发布 `/scan`（点云扫描数据）
2. AMCL 读取 `/scan` 和 `/odom`，用粒子滤波估计机器人在地图中的位置，
   发布 `/amcl_pose`（带协方差的位姿估计）以及 `map→odom` 的 TF 变换
3. 多点导航节点向 move_base 发送 `/move_base/goal`（目标位姿，在 `map` 坐标系下）
4. move_base 通过 TF 获知机器人当前在地图中的位置（`map→odom→base_footprint`），
   利用全局代价图规划一条从当前位置到目标的全局路径
5. 局部规划器（TEB）在行进过程中实时调整速度指令，避开动态障碍物，
   将速度指令通过 `/cmd_vel` 发送给底盘驱动
6. 机器人到达目标后，move_base 返回 SUCCEEDED 状态给导航节点

## 2. 坐标系与 TF 树

ROS 用 TF（Transform Library）管理不同坐标系之间的变换关系。
本系统涉及的坐标系及其含义如下：

| 坐标系名称         | 描述                                                      | 发布者   |
|--------------------|-----------------------------------------------------------|----------|
| `map`              | **地图坐标系**。原点是建图时机器人的起始位置和朝向。        | AMCL     |
| `odom`             | **里程计坐标系**。根据轮式编码器积分推算，会随时间漂移。    | 底盘驱动 |
| `base_footprint`   | **机器人足迹坐标系**。机器人在地面的投影中心。              | 底盘驱动 |
| `base_link`        | **机器人本体坐标系**。与 `base_footprint` 固连。            | 底盘驱动 |
| `laser`            | **激光雷达坐标系**。激光扫描数据参考系。                    | 底盘驱动 |

**TF 树结构：**

```
map
 └── odom          （AMCL 发布，表示"地图定位修正"）
      └── base_footprint  （底盘里程计发布）
           └── base_link
                └── laser
```

**关键理解：**

- `odom→base_footprint` 由底盘里程计发布，精度高但会漂移（短期准确，长期漂移）
- `map→odom` 由 AMCL 发布，是 AMCL 对"里程计漂移量"的修正
- **若 AMCL 未就绪（未发布 `map→odom`），整个 TF 树从 `map` 往下的变换
  将不可用，move_base 无法获知机器人在地图中的位置，导航会失败或产生严重偏差**

## 3. AMCL 自适应蒙特卡洛定位

### 3.1 工作原理

AMCL（Adaptive Monte Carlo Localization，自适应蒙特卡洛定位）使用**粒子滤波**
在已知地图中估计机器人的位置。

```
初始化：在地图上撒 N 个"粒子"，每个粒子代表一个可能的机器人位姿 (x, y, θ)
         ↓
运动更新：当里程计显示机器人移动时，所有粒子按相同运动模型移动（加入噪声）
         ↓
观测更新：获得激光扫描数据后，计算每个粒子位置的"观测似然度"
         （即：如果机器人真的在粒子的位置，能不能看到这样的激光扫描结果？）
         ↓
重采样：权重高的粒子被复制更多，权重低的被淘汰
         ↓
输出：取粒子的加权均值作为最优位置估计，发布 /amcl_pose 和 map→odom TF
         ↓
         （持续循环）
```

### 3.2 为什么初始位姿至关重要

粒子滤波的收敛速度取决于初始粒子分布的质量：

- **初始位姿正确**：粒子集中在真实位置附近 → 少量激光扫描即可收敛 → 几秒内就绪
- **初始位姿错误**：粒子分散在错误区域 → 需要大量激光数据才能"跳"到正确位置
  （有时根本无法自动收敛，机器人必须移动一段距离让 AMCL 观察到更多特征）

**默认初始位姿是地图坐标原点 (0, 0, 0)**，这正是建图时机器人的起始位置。
每次导航前，机器人应放回**与建图时完全相同的位置和朝向**，或者通过以下方式之一
指定初始位姿：

1. **RViz "2D Pose Estimate"**：在 RViz 界面中手动点击机器人当前位置并拖拽方向
2. **`set_initial_pose` 参数**：节点启动时自动向 `/initialpose` 发布位姿（程序方式）

### 3.3 如何判断 AMCL 已收敛

在 RViz 中订阅 `/particlecloud` 话题（绿色箭头粒子云）：

- **未收敛**：粒子分散在地图的大范围区域，或集中在错误位置
- **初步收敛**：粒子基本聚集在一个较小区域，形状较长（位置确定但方向不确定）
- **收敛良好**：粒子紧密聚集成一个小团，方向一致 ← **此时才适合开始导航**

也可以查看终端：本节点在等待 AMCL 时会打印：
```
✓ AMCL 定位已就绪，可以开始导航。
```

### 3.4 常见 AMCL 问题

| 症状 | 可能原因 | 解决方法 |
|------|----------|----------|
| 导航路径奇怪，终点偏差大 | AMCL 未收敛就开始导航 | 等待粒子云收敛后再启动，或使用 `set_initial_pose` |
| 机器人越走越偏 | 激光雷达被遮挡或环境变化 | 确认雷达视野清晰，重新标定 |
| 机器人突然"跳位" | AMCL 重新定位（全局重定位） | 检查地图与实际环境是否一致 |
| 始终无法收敛 | 初始位姿偏差太大 | 手动用 RViz "2D Pose Estimate" 标注 |

## 4. move_base 导航框架

### 4.1 接收目标的方式

move_base 通过 actionlib 接口接收导航目标（`MoveBaseGoal`），目标包含：
- `target_pose.header.frame_id`：目标所在的坐标系（通常为 `"map"`）
- `target_pose.pose.position.x/y`：目标位置
- `target_pose.pose.orientation`：目标朝向（四元数）

### 4.2 执行流程

```
收到目标
   ↓
清除代价地图（可选）
   ↓
全局规划器：在全局代价地图上规划从当前位置到目标的路径
   ↓
路径可达？ ─── 否 ──> 返回 ABORTED（无法规划路径）
   ↓ 是
局部规划器循环：
   ├── 读取当前位置（via TF: map→base_footprint）
   ├── 读取激光扫描（更新局部代价地图）
   ├── 计算速度指令并发布到 /cmd_vel
   ├── 到达目标？ ─── 是 ──> 返回 SUCCEEDED
   └── 超时或卡住？ ─ 是 ──> 尝试恢复行为 → 失败则返回 ABORTED
```

### 4.3 目标到达的判断标准

move_base 认为"到达目标"需同时满足：

- 位置误差 < `xy_goal_tolerance`（默认约 0.15 m）
- 角度误差 < `yaw_goal_tolerance`（默认约 0.1 rad ≈ 5.7°）
- 速度接近零：`trans_stopped_vel` 和 `rot_stopped_vel`

若 `xy_goal_tolerance` 设置过大（如 0.5 m），机器人会在距目标较远处就停止，
表现为"终点偏差大"（但路径正常）。

### 4.4 恢复行为

当机器人被困（无法前进）时，move_base 会尝试内置的恢复行为，默认顺序为：

1. 清除局部代价地图的障碍层 → 重新规划
2. 旋转到位 → 重新规划
3. 清除全局代价地图的障碍层 → 重新规划
4. 所有恢复失败 → 返回 ABORTED

## 5. 代价地图（Costmap）

代价地图是 move_base 对环境的数字化表示。每个栅格（grid cell）的值表示
"通过该位置的代价"：0 = 完全自由，254 = 致命障碍，255 = 未知区域。

### 5.1 全局代价地图（Global Costmap）

- 基于静态地图（由地图服务器加载的 `.pgm` 地图文件）构建
- 包含整个地图范围，用于全局路径规划
- 启动后基本不变，但 AMCL 位姿更新会影响机器人在地图中的起点

### 5.2 局部代价地图（Local Costmap）

- 以机器人为中心的滑动窗口（如 3m×3m）
- 实时根据激光扫描数据更新，能感知静态地图中没有的动态障碍物
- 障碍物会根据 `inflation_radius` 参数进行膨胀（留出安全距离）

### 5.3 代价地图与路径偏差的关系

当 AMCL 定位不准时，代价地图中机器人起点是错误的：

```
地图坐标系：
  机器人实际位置：(1.0, 1.0)
  AMCL 误认的位置：(0.5, 0.5)   ← AMCL 未收敛
  目标位置：(2.0, 2.0)
  
  AMCL 误认时：规划路径为 (0.5,0.5)→(2.0,2.0)，机器人实际执行的起点是
              (1.0,1.0)，路径方向相同但起点不同，导致终点约在 (2.5,2.5)
              而非 (2.0,2.0)。
```

## 6. 路径规划器

### 6.1 全局规划器

默认使用 **navfn**（Dijkstra/A* 算法）在全局代价地图上搜索最短可行路径。
规划结果是一系列 `(x, y)` 坐标点组成的参考路径，不含速度信息。

本项目 `navigation.launch` 中配置使用 **TEB（Time Elastic Band）** 作为局部规划器，
同时 TEB 也可作为全局规划器。

### 6.2 局部规划器：TEB（时间弹性带）

TEB 将机器人的运动轨迹表示为一系列带时间戳的位姿，通过最优化使轨迹满足：

- **路径跟踪约束**：轨迹贴近全局规划路径
- **运动学约束**：差速驱动底盘的速度/加速度限制
- **障碍物约束**：与局部代价地图中的障碍物保持安全距离
- **时间最优**：在满足约束的前提下最小化行驶时间

**TEB 对差速驱动小车（本项目使用三轮差速）的适用性：**
TEB 原生支持差速驱动（`holonomic_robot = false`），能生成符合差速约束的轨迹。

## 7. 多点导航脚本工作流程

### 7.1 启动流程

```
启动节点
   ↓
读取 ROS 参数（loop, timeout, 坐标系, 旋转阈值, 初始位姿等）
   ↓
set_initial_pose=true？ ─── 是 ──> 等待 1 秒 → 发布 /initialpose → 等待 1 秒
   ↓
等待 AMCL 就绪（最多 amcl_wait_timeout 秒）
   ↓
AMCL 未响应？ ─── 是 ──> 打印警告（但继续，不强制退出）
   ↓
连接 move_base 动作服务器（最多 30 秒）
   ↓
开始执行航点列表（见 7.2）
```

### 7.2 单个航点执行决策

```
计算 dist = |当前航点 XY − 上一位置 XY|
             ↑ "上一位置" 优先取 TF 实际位置，TF 失败时取上一航点目标坐标
   ↓
dist < rotate_in_place_threshold (默认 0.05 m)？
   ├── 是：
   │     直接发布 /cmd_vel 角速度，原地旋转到目标 yaw
   │     （绕过 move_base，避免 XY 漂移）
   │
   └── 否：
         构造 MoveBaseGoal（map 坐标系下的目标位姿）
         发送给 move_base 动作服务器
         等待结果（最多 goal_timeout 秒）
         状态 == SUCCEEDED？ ──否──> 报错，停止导航
               ↓ 是
         成功，继续
   ↓
通过 TF 读取实际到达位置，更新 prev_x/prev_y
   ↓
interactive=true 且还有下一个航点？ ──> 显示交互提示
```

### 7.3 航点坐标的来源

航点坐标（`x`, `y`, `yaw`）应从 RViz 中采集：

1. 启动 RViz 并加载导航可视化
2. 在顶部工具栏点击 "2D Nav Goal"
3. 在地图上点击目标位置并拖拽确定方向
4. 终端会打印目标位姿信息，记录其中的 `x`, `y` 以及从四元数换算的 `yaw`

```
# RViz 2D Nav Goal 在 /move_base_simple/goal 话题上发布目标
# 可以用 rostopic echo 记录：
rostopic echo /move_base_simple/goal
```

## 8. 已知问题与解决方案

### 问题 1：同位置旋转后 XY 坐标产生偏移

**现象：** 航点只有 `yaw` 变化（`x/y` 不变），小车旋转完成后，实际位置在 X/Y 上
也出现了偏差（如 1~5 cm 的漂移）。

**原因：** 若直接将"只有角度变化的航点"发给 move_base，move_base 为同时满足
`xy_goal_tolerance` 和 `yaw_goal_tolerance`，规划器会让底盘做微小的前后移动，
这些移动通过轮式里程计传播到 AMCL 位姿，累积后产生可见偏差。

**解决方案（已实现）：**
当 `dist(当前 XY, 目标 XY) < rotate_in_place_threshold`（默认 0.05 m）时，
节点绕过 move_base，直接向 `/cmd_vel` 发布纯角速度指令，实现零 XY 位移的原地旋转。

```bash
# 若需要调整阈值或旋转速度，在 launch 时设置：
roslaunch wulina multi_point_navigation.launch \
  rotate_in_place_threshold:=0.05 \
  angular_speed:=0.3 \
  yaw_tolerance:=0.05
```

---

### 问题 2：AMCL 初始位姿与建图起点不一致

**现象：** 导航开始后，机器人的路径和最终位置相对于目标有系统性的固定偏移。

**原因：** AMCL 默认初始位姿为地图原点 `(0, 0, 0)`，即建图时机器人所在的位置。
若导航时机器人被放到不同的位置，AMCL 粒子从错误起点收敛，需要一段时间才能
纠正，期间所有导航目标都会有系统偏差。

**解决方案（已实现）：**

**方法 A（推荐，通用）：** 使用 RViz "2D Pose Estimate" 工具，在地图上点击机器人
当前位置并指定方向。AMCL 会立即重新初始化粒子，通常几秒内收敛。

**方法 B（程序化，适合固定起点）：** 若每次都在相同的非原点位置出发，设置：
```bash
roslaunch wulina multi_point_navigation.launch \
  set_initial_pose:=true \
  initial_pose_x:=1.23 \
  initial_pose_y:=0.45 \
  initial_pose_a:=0.785
```
节点启动时会自动向 AMCL 发布初始位姿，无需手动操作 RViz。

---

### 问题 3：改变 XY 时终点偏差大，路径奇怪

**现象：** 当目标点的 `x/y` 发生变化时（即使角度不变），机器人走的路径与预期
不符，最终停靠位置与目标有明显偏差（可能超过 0.5 m）。

**原因分析：**

| 原因 | 典型表现 | 判断方法 |
|------|----------|----------|
| AMCL 未收敛就开始导航 | 路径完全偏离，终点远离目标 | RViz 中粒子云散布；`/amcl_pose` 话题未发布 |
| `goal_tolerance` 过大 | 路径正常，但提前停止（差 0.3~0.5 m） | 调小 `xy_goal_tolerance` |
| 地图与实际环境不符 | 路径在地图中正确，现实中绕远 | 对比地图与实际障碍物 |
| TF 树断裂（AMCL 崩溃） | move_base 报 TF 错误，无法规划 | `rostopic echo /tf_static` 检查变换 |

**解决方案（已实现）：**

1. **等待 AMCL 就绪**：节点启动时自动等待 `/amcl_pose` 发布消息，确认 AMCL
   已激活后再开始发送导航目标，避免从未收敛的位置开始规划：

   ```bash
   # 默认等待 30 秒；若 AMCL 在 30 秒内发布了位姿，立即继续
   # 若需要更长时间，可以增大：
   roslaunch wulina multi_point_navigation.launch amcl_wait_timeout:=60.0
   # 若 AMCL 已确认就绪（如自动化测试），可跳过等待：
   roslaunch wulina multi_point_navigation.launch amcl_wait_timeout:=0   ```

2. **用 TF 实际位置更新 prev_x/prev_y**：每个航点完成后，代码读取 TF
   中机器人的真实坐标，而非假设精确停在目标坐标。这确保后续旋转/移动
   决策基于真实状态，减少累积误差。

**排查步骤：**

```bash
# 步骤 1：确认 AMCL 在发布位姿
rostopic echo /amcl_pose --n 1

## 9. 调参建议

### 9.1 与导航精度最相关的参数

下表列出对导航精度影响最大的参数，按重要性排序：

| 参数 | 所在文件 | 说明 | 推荐值（差速小车） |
|------|----------|------|-------------------|
| `xy_goal_tolerance` | `move_base.yaml` 或 `teb_local_planner_params.yaml` | 位置到达容差（米） | 0.10~0.20 |
| `yaw_goal_tolerance` | 同上 | 角度到达容差（弧度） | 0.05~0.10 |
| `min_obstacle_hdist` / `inflation_radius` | `costmap_common_params.yaml` | 机器人与障碍物的最小安全距离 | 机器人半径 + 0.05 m |
| `max_vel_x` | `teb_local_planner_params.yaml` | 最大前进速度（m/s） | 0.3~0.5 |
| `max_vel_theta` | 同上 | 最大旋转速度（rad/s） | 0.5~1.0 |

### 9.2 AMCL 相关参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `min_particles` | 最少粒子数 | 100（速度）~500（精度） |
| `max_particles` | 最多粒子数 | 2000~5000 |
| `update_min_d` | 触发滤波更新的最小移动距离（米） | 0.1~0.2 |
| `update_min_a` | 触发滤波更新的最小旋转角度（弧度） | 0.2~0.5 |
| `laser_max_range` | 激光扫描最大有效距离（米） | 与实际雷达参数一致 |
